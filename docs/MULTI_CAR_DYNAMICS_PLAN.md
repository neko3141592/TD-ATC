# 編成車両別・運動力学計算の実装計画

## 1. 背景と目的

現在の物理モデルでは、列車全体を「1つの点（先頭車の位置）」として扱い、先頭車がいる位置の勾配やカーブの条件を編成全体に適用しています。
しかし、実際の鉄道では編成が長いため、次のような現象が起こります。

* **勾配の進入・進出時のタイムラグ**: 先頭車が上り勾配に入っても、最後尾の車両が平坦な場所にいれば、編成全体が受ける勾配抵抗は徐々に増加します（急激に減速しない）。
* **曲線抵抗の分散**: 編成の一部だけが急カーブに入っている場合、カーブによる減速効果（曲線抵抗）は編成全体に分散されます。
* **M車/T車の配置による力の分布**: 動力車（M車）と付随車（T車）の位置によって、編成内にかかる引張力・圧縮力が変わります。

これを解決するため、**「全車両の現在位置を個別に計算し、それぞれの車両が受ける力（牽引力・制動力・勾配抵抗など）を合算して、編成全体の加速度を求める」**モデルへと移行します。これを「車両分散配置の剛体モデル」と呼びます。

---

## 2. 期待される効果

1. **リアルな走行フィーリング**: 勾配の変わり目やカーブの出入りで、加速度の変化が非常に滑らかで実感に近くなります。
2. **正確な走行抵抗計算**: 長大編成になるほど顕著になる、地形変化に対する「重さ」と「粘り」が表現できます。
3. **拡張性の向上**: 将来的に、連結器のガタつき（衝撃）や、車両ごとの脱線判定などを実装する際の確固たる土台になります。

---

## 3. 実装のステップ（フェーズ分割）

連結器のバネやダンパ（伸び縮み）までシミュレートすると計算が非常に複雑になり発散（暴走）しやすくなるため、まずは**「全車両は常に同じ速度・同じ加速度で動く（連結器は伸び縮みしない剛体）」**という前提で実装します。

### Phase 1: 各車両の軌道上位置の算出
* 現在は `TrainController` が先頭車の `distanceOnEdgeM` だけを持っていますが、各車両が「先頭車から何メートル後ろにいるか（オフセット）」を定義します。
* 例: 1両目 = `0m`, 2両目 = `-20m`, 3両目 = `-40m` ...
* 先頭車の距離からオフセットを引き、各車両ごとの現在の `TrackEdge` とその中の `distance` を逆算するロジック（`TrainRoutingState` の派生など）を作ります。

### Phase 2: 車両個別の「力（N）」の計算
`ExternalForceCalculator.cs` 等を拡張し、各車両が個別に力を計算できるようにします。
1. **勾配抵抗 ($F_{grad}$)**: 各車両がいる位置のパーミル値を取得し、$m \times 9.8 \times (\text{permille} / 1000)$ を計算。
2. **曲線抵抗 ($F_{curve}$)**: 各車両がいる位置のカーブ半径を取得し、抵抗を計算。
3. **走行・空気抵抗 ($F_{run}$)**: 各車両の形状や重さに応じた抵抗。先頭車だけは空気抵抗を大きくするなどの工夫も可能。
4. **牽引力/制動力 ($F_{motor} / F_{brake}$)**: すでに実装されている `CarBrakeState` や `CarTractionState` を利用して、M車やT車ごとの出力を得る。

### Phase 3: 運動方程式の統合（ニュートンの第2法則）
各車両ごとの力 $F_i$ を、編成全体で合計します。

$$ F_{total} = \sum_{i=1}^{N} (F_{motor, i} - F_{brake, i} - F_{run, i} - F_{grad, i} - F_{curve, i}) $$

編成の総質量 $M_{total}$ で割ることで、編成全体の加速度 $a$ が求まります。

$$ a = \frac{F_{total}}{M_{total}} $$

求めた加速度 $a$ で速度を更新し、全車両をその速度で移動させます。

### Phase 4: 3Dモデルの正確な配置
算出された各車両の「トラック上の正確な距離（distance）」を使って、`TrackRuntimeResolver.cs` で1両ずつ位置と角度（X, Y, Z, Pitch, Yaw）を計算し、3Dモデルを配置します。

---

## 4. 懸念点と課題

* **エッジのまたぎ処理の複雑化**: 先頭車が次の `TrackEdge` に進んでも、後ろの車両はまだ前の `TrackEdge` に残っている状態の管理が必要です。エッジの繋がり（履歴）をキャッシュしておくか、後続車が常に先頭車の通ったルートをトレースする仕組み（パンくずリスト方式）が必要です。
* **処理負荷**: 車両数ぶんだけ、毎フレーム `TrackRuntimeResolver.cs`の計算が走ります。最大でも10〜15両程度であれば現在のPC/スマホの性能なら全く問題ありませんが、計算効率は意識する必要があります。

---

## 5. 次にとるべきアクション

この計画に同意できる場合、まずは **Phase 1 と Phase 4（各車両を軌道上に等間隔で配置し、先頭に追従させる仕組み）** から実装を始めるのが最も安全かつ視覚的にもわかりやすい第一歩となります。
