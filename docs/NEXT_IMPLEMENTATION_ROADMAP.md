# 今後の実装ロードマップ

作成日: 2026-04-17

## 目的

現在の実装は「単純な距離軸の運転ゲーム」から「線路グラフ上を走る鉄道運転シミュレーション」へ移行中です。

今後は、次の順で積み上げます。

1. 線路グラフ走行を安定させる
2. 行路・停車駅・停車判定を入れる
3. 閉塞在線を入れる
4. ATCを閉塞・分岐対応へ拡張する
5. TASCとゲーム進行を追加する
6. AI列車・折返し・音まわりを仕上げる

## 現在の到達点

### 走行・車両

- `TrainSpec` に力行、制動、回生、空気ブレーキ、走行抵抗の主要パラメータが入っている。
- `TrainController` は速度、加速度、累計距離、現在エッジID、エッジ内距離を管理している。
- `NotchManager` で手動、ATC、TASC のブレーキ指令を統合する入口ができている。
- `ConsistDefinition` / `CarSpec` / `CarBrakeState` / `CarTractionState` により、編成単位の質量、力行、回生、空気ブレーキを扱える。
- `BrakeSystemController` は M車回生、T車空気、M車空気、非常ブレーキを分けて計算している。
- `TractionSystemController` はM車のモータ数に応じて力行力を配分している。

### 線路グラフ

- `TrackGraph` に `TrackNode`、`TrackEdge`、`TurnoutState`、`StationData` が入っている。
- `TrackEdge` は曲線レシピ `mathCurves`、長さ、速度制限を持っている。
- `TrackRuntimeResolver` は直線、円曲線、緩和曲線、勾配から、エッジ上の位置と接線を解決できる。
- `TrainController` はエッジ終端を超えたら `TrackGraph.ResolveNextEdgeId()` で次エッジへ遷移できる。
- 編成後方位置用に `activeEdges` と `CarTrackState` があり、車両ごとのエッジ内位置を追える。
- `TrackVisualizer` / `TrackMeshGenerator` により、グラフからレールと枕木のメッシュを生成できる。

### ATC・UI・音

- `ATCController` は現在エッジの速度制限を読み、先の低速エッジに向けた許容速度パターンを生成できる。
- `ATCPatternCalculator` は `v_allow = sqrt(v_target^2 + 2ad)` の計算を分離済み。
- 超過時はATCブレーキをラッチし、解除マージンで戻す仕組みがある。
- HUD は速度、距離、加速度、ノッチ、ATC指令、制動力、BC圧、車両別ブレーキ状態を表示できる。
- `Compressor` と `CarBuzzer` の音コンポーネントがある。

### 着手中の変更

- `TrainServiceDefinition` と `ServiceStop` が追加され、列車種別ごとの停車/通過を定義する入口ができている。
- `NextStationResolver` が追加され、現在エッジ上の次停車駅を取得できる。
- `NextStationResolver` は現時点では同一エッジ上の駅だけを見るため、次フェーズで複数エッジ先まで辿る必要がある。
- `CabBuzzer` から `CarBuzzer` へのクラス名修正が入っている。

## 主要な未実装・リスク

- 閉塞定義と在線管理がない。
- `TrackEdge` に閉塞IDがまだないため、ATCは先行列車ではなく速度制限だけを見ている。
- 複数エッジ先を一定距離まで辿る汎用APIがない。
- 次駅解決は同一エッジ限定で、分岐、通過駅、ルート順序にはまだ対応していない。
- 停車成立判定、ドア扱い、発車条件、ダイヤ時刻がない。
- TASC用の停止パターン生成とノッチ指令がまだない。
- AI列車、折返し、対向列車、進路予約は未実装。
- Unity上の自動テストはまだなく、グラフ検証はエディタボタンでの手動実行が前提。

## Phase 1: 線路グラフ基盤の安定化

目標: 1列車が複数エッジを跨いでも破綻せず、後続機能が同じ経路情報を使える状態にする。

状態: 実装完了（2026-04-19）

実装済み:

- `TrackTraceSegment` と `TrackRouteTracer` を追加。
- 現在エッジから指定距離先まで、開通中の進路に沿ったエッジ区間列を取得可能。
- `TrainController` の終端到達時は速度と加速度を0にして、線路終端で停止。
- `activeEdges` は先頭車の現在エッジを先頭に、後方車両に必要な履歴だけ保持。
- `TrackGraph.ValidateGraph()` を追加。
- `TrackGraphEditor` に `Validate Graph` ボタンを追加。
- 旧距離軸ATC定義の `ATCProfile` / `SpeedBlock` と未使用assetを削除。

実装タスク:

- `TrackRouteTracer` を追加し、現在エッジから指定距離先までのエッジ列を取得できるようにする。
- 返却データは `edgeId`、エッジ内開始距離、エッジ内終了距離、現在地からの累積距離を持たせる。
- `TrainController` のエッジ遷移と `activeEdges` 更新を、経路トレースと矛盾しないよう整理する。
- 終端到達時の挙動を決める。まずは停止、将来は折返しまたは車止め扱い。
- `TrackGraph` に検証メソッドを追加する。
  - node/edge ID重複
  - edgeのfrom/to存在
  - outgoingEdgeIdsの整合
  - junctionの選択edge妥当性
  - lengthMとmathCurves合計の差

完了条件:

- 直線、曲線、分岐を含むグラフで、1列車が複数エッジを連続走行できる。
- 経路トレースAPIを ATC、次駅、閉塞の共通入力として使える。
- 不正な `TrackGraph` をエディタ上で検出できる。

## Phase 2: 行路・停車駅・停車判定

目標: 列車種別ごとに次停車駅を決め、停車成功/失敗を判定できるようにする。

実装タスク:

- `TrainServiceDefinition` を駅順序付きにする。
  - `serviceName`
  - `serviceType`
  - 停車駅リスト
  - 停車/通過
  - 標準停車時間
  - 任意で到着/発車目標時刻
- `NextStationResolver` を複数エッジ対応に拡張する。
  - `TrackRouteTracer` の結果を使う。
  - 同一エッジだけでなく、進路上の駅を順に見る。
  - 通過駅はスキップする。
  - 進路が分岐した場合は現在の `TurnoutState` に従う。
- `StationStopController` を追加する。
  - 停止位置までの距離
  - 停止許容誤差
  - 停止判定速度
  - 停止維持時間
  - 次駅インデックス進行
- HUDに次駅名、次駅距離、停車/通過、停止誤差を追加する。

完了条件:

- 1路線で次停車駅が正しく更新される。
- 通過駅を通過しても次停車駅が崩れない。
- 停止位置の前後誤差と停車成功/失敗が安定して出る。

## Phase 3: 閉塞在線管理

目標: 先行列車がいる閉塞までの距離を取得し、ATCの入力にできる状態にする。

実装タスク:

- `TrackEdge` に `blockId` を追加する。最初は1エッジ1閉塞でよい。
- 必要なら `BlockDefinition` を追加する。
  - `blockId`
  - 関連edgeIds
  - 停止基準余裕距離
  - 閉塞終端または停止目標点
- `BlockOccupancyManager` を追加する。
  - blockIdごとの在線列車ID集合
  - 列車ごとの現在占有block集合
  - enter/exit差分更新
- 占有判定は最初は先頭車位置だけで開始し、次に編成全体の `CarTrackState` を使う。
- `TryFindFirstOccupiedBlockAhead()` を追加する。
  - 現在エッジ
  - エッジ内距離
  - 探索距離
  - 自列車ID
  - 結果: blockId、距離、目標速度0

完了条件:

- 2列車を同一進路に置くと、後続列車から先行在線閉塞までの距離が取れる。
- 自列車の占有を誤検出しない。
- 分岐器の開通方向が変わると探索先も変わる。

## Phase 4: ATC v2

目標: 速度制限だけでなく、先行在線閉塞にも反応するATCにする。

実装タスク:

- `ATCController` の目標候補を2系統にする。
  - 候補A: 進路上の次の低速制限
  - 候補B: 先行在線閉塞の停止目標
- 両候補から許容速度を計算し、より厳しい値を採用する。
- `TrackRouteTracer` を使い、分岐後の進路に沿って探索する。
- `safetyDistance` を候補ごとの停止余裕として扱う。
- HUDに以下を追加する。
  - 現在制限
  - パターン許容速度
  - ATC目標種別
  - ATC目標距離
  - ATCブレーキラッチ状態

完了条件:

- 先行列車ありで後続列車に停止パターンが出る。
- 先行列車なしでは不要な制限が出ない。
- 低速制限と在線閉塞が同時にある場合、厳しい方が採用される。
- ATCがTASCや手動入力より安全側で優先される。

## Phase 5: TASC最小版

目標: 次停車駅の停止位置へ自動停止補助をかける。

実装タスク:

- `TASCController` を追加する。
- `StationStopController` または `NextStationResolver` から停止目標距離を受け取る。
- 停止目標速度0へ向けて許容速度を計算する。
- 現在速度が許容速度を超えたら `NotchManager.SetTASCBrakeNotch()` でブレーキ指令を出す。
- 停止直前は低速域専用の微調整を入れる。
  - 低速失効した回生を前提に空気ブレーキ中心で調整
  - 過走しそうなら高め、手前停止しそうなら緩める
- ATCが作動している場合はATCを優先する。

完了条件:

- 同じ初期条件で5回走らせて、停止誤差が許容範囲内に収まる。
- TASC作動中でもATCブレーキが上書きできる。
- 停車後、TASC指令が解除される。

## 直近でやるべき順番

### Task 1: `TrackRouteTracer` を作る

理由:

- ATC、次駅、閉塞、TASCの全てが「進路上の距離」を必要とするため。
- ここを共通化しないと、それぞれが別々にエッジ探索を実装して手戻りが出る。

最小API案:

```csharp
public bool TryTraceAhead(
    TrackGraph graph,
    string currentEdgeId,
    float distanceOnEdgeM,
    float lookaheadDistanceM,
    List<TrackTraceSegment> results
)
```

### Task 2: `NextStationResolver` を複数エッジ対応にする

理由:

- 現在の `NextStationResolver` は同一エッジ上の駅しか見ない。
- 停車駅、TASC、ゲーム進行の前提になるため、早めに直す必要がある。

### Task 3: `StationStopController` を作る

理由:

- TASCより先に、手動運転で「停車できたか」を判定できる必要がある。
- 停車判定がないと、TASCの良し悪しを評価できない。

### Task 4: `blockId` と `BlockOccupancyManager` を入れる

理由:

- ATC v2の中核。
- 先行列車を検知できるようになると、ゲームとしての運転制約が一気に成立する。

### Task 5: ATC v2へ差し替える

理由:

- 現在のATCは速度制限パターンとしては動くが、在線閉塞を見ていない。
- 閉塞が入った後に差し替えると、設計の手戻りが少ない。
